Projet de rattrapage

Dans le cadre du HACKATHON M1 AL, ce projet s'inscrit dans les cas spécifiques nécessitant un rattrapage car l'élève n'a pas pu être présent pour raisons justifiés lors du HACKATHON.

Objectif(s) pédagogique(s)

Ce projet vise à aborder tous les pans techniques d'un projet applicatif s'inscrivant dans un système d'entreprise. On y abordera donc de l'architecture logicielle (microservices, hexagonale) et d'infrastructure (api gateway, docker swarm), de la sécurité (authentification), de la CI/CD (git workflow), ainsi que de la supervision d'applicatifs (service discovery).

Contrainte(s)

Language Backend & Frontend libre Architecture backend logicielle : Architecture Hexagonale Architecture d'infrastructure : Docker Swarm Repository : Github API Gateway : NGINX CI/CD : Git Workflow Auth : JWT Supervision : Prometheus

Evaluaton

Soutenance

Rendu

  • Documentation technique (format libre; ex : mkdoc)

Besoin(s)

L'entreprise X a besoin d'une application pour aider ses employés à gérer leurs temps et leurs tâches. Ils souhaitent faire un applicatif simple mais en interne pour gérer les données potentiellement confidentielles que ces tâches pourraient contenir.

Ils souhaitent que cette application soit accessible par compte utilisateur et qu'elle soit le plus simple possible.

Les tâches auront donc un nom, une date de création, une date effective et une description. Chaque tâche est assignée à son créateur.

Architecture technique

Le système applicatif mettra en place une architecture microservice, chaque composant de l'application fait une seule chose et dans l'absolu aurait une seule base de données (ce qui n'est pas forcément le cas dans la pratique).

API Gateway

L'API Gateway sert à aiguiller les requêtes vers l'API concerné, son rôle est d'offrir au client un seul point d'entrée et donc d'ajouter une couche d'abstraction réseau.

API Authentification

L'API Authentification contient toute la logique d'autentification par token et sera sollicité par l'API Gateway à la réception des requêtes.

API CRUD

Une API CRUD sert à réaliser des actions basiques (création, modification, suppression et lecture) sur une donnée.

API CRUD - USER & TASK

Cette API sert à réaliser des actions CRUD sur la ressource USER et TASK.

On regroupe ici les deux API pour faciliter le développement. Dans un autre contexte, ces deux APIs pourraient être séparées.

Première partie - Mise en place

Repository

Mettez en place un dépôt Github par partie de votre applicatif.

Partagez vos dépôts ou un dépôt principal (organisation) à votre référent pédagogique par mail.

Choix des technologies

Choisissez les technologies pour le backend et le frontend.

Documentez les justifications de vos choix. (README ou MkDoc)

CI/CD

Mettez en place votre Github registry pour stocker les images docker que vous allez créer. Mettez en place une Github action pour chaque dépôt qui testera la compilation de chacun des projets. Mettez en place une Github action pour chaque dépôt qui poussera une nouvelle image docker vers le Github registry. Mettez en place un DockerFile par partie de votre applicatif pour en créer des images. Mettez en place un docker-compose pour récupérer les images depuis le registry Github et démarrer les conteneurs.

Documentez le système que vous venez de mettre en place. (README ou MkDoc)

Deuxième partie - API Gateway

Mettez en place un serveur web NGINX pour rediriger des requêtes vers l'application souhaité selon l'url.

Utilisez le système de upstream.

Mettez en place la redirection vers l'API d'authentification afin de sécuriser l'accès aux autres url. Mettez en place ce qu'il faut pour que votre serveur web soit déployé en tant que conteneur.

Documentez la configuration et le système mis en place.

Troisième partie - Développement des APIs

API AUTH

Note : Organisez votre code selon une Architecture Hexagonale.

Mettez en place une API qui va vous permettre de : - Vérifiez votre utilisateur et récupérez un token - Vérifiez l'existence et la validité d'un token (expiration, signature) - Récupérez le rôle assigné à l'utilisateur et l'injecter en tant que claim dans le token.

Si les données utilisateurs sont nécessaires alors favoriser de requêter directement l'API USER. (ex: WebClient en Java).

API CRUD USER

Note : Organisez votre code selon une Architecture Hexagonale.

Mettez en place vos endpoints CRUD pour la ressource USER. Mettez en place vos endpoints CRUD pour la ressource TASK.

Documentez vos APIs avec OpenAPI/Swagger. Résumer vos APIs (Données entrante, Réponse, codes erreur traités, etc...) dans votre README ou MkDoc.

Mettez en place un système d'interception des requêtes HTTP (ex : Filter en Java) afin de récupérer le rôle contenu dans le token JWT. - Il existe deux rôles STANDARD et ADMIN. - Un administrateur peut agir au niveau de la gestion des utilisateurs. - Un utilisateur qui a le rôle STANDARD, n'a accès qu'à la gestion de ses tâches.

Complétez votre documentation. Expliquez comment vous avez géré les rôles qui sont partagés entre les APIs CRUD et l'API AUTH.

Quatrième partie - IHM

Mettez en place une interface simplifié d'affichage des tâches par colonne : - Passées : la date de la tâche est passée - En cours : la date de la tâche est la date du jour - A venir : la date de la tâche est à venir.

Mettez en place une interface admin qui permets de gérer les utilisateurs.

Note : La partie client ne devra exclusivement passé que par l'API Gateway.

Cinquième partie - Swarm & Supervision

Docker Swarm

Mettez ce que vous avez fait sous docker-compose au sein d'un Docker Swarm.

Documentez votre architecture Docker Swarm dans votre README ou MkDoc.

Documentez les mesures à prendre pour rendre votre architecture résiliente à la charge et à la panne.

Supervision

Mettez en place un Prometheus afin de superviser les informations des conteneurs.

Documentez ce que vous avez dûe mettre en place et configurer.
