HACKATHON

Intervenant : MARIN Geoffrey

Sujet

Mise en oeuvre d'une architecture microservices polyglotte conteneurisée pour répondre au besoin applicatif d'un CE d'entreprise.

Contrainte(s)

L'équipe du projet sera formé de 4 groupes de 4 personnes, chaque groupe :

Le projet est commun à tous les groupes; chaque groupe sera en charge d'une ou plusieurs parties de l'application.

Exemple de découpage d'application en plusieurs parties : IHM (Frontend), API Utilisateur (Backend #1), API Tâches (Backend #2), API Tableaux (Backend #3), Base de données, DevOps (CI/CD, Docker), Documentation (mkdocs, javadoc, swagger, etc...), Présentation (canvas, ppt, etc...), Recette (test).

Le choix des technologies de développement est globalement libre mais adapté et justifié.

Technologie(s)/Outil(s) imposé(e)(s): - Docker/Docker Swarm - GitHub

Proposition de technologie(s) : - Documentation : MkDocs (MarkDown)

Rendu

  • Documentation Architecture Technique :
    • Format : Un document PDF
    • Contenu :
      • Schéma architecture technique détaillé
      • Procédure de lancement de l'architecture microservices.
  • Documentation par groupe :
    • Format : Lien Git Pages (mkdocs) dans le README ou un document PDF. (Attention à la mise en page)
    • Contenu :
      • Explication du contexte applicatif (place occupée au sein de l'architecture) : texte + schéma
      • Définition de la fonctionnalité implémentée.
      • Schéma de base de données.
      • Explication de l'implémentation technique.
      • Procédure de mise en place et de lancement du service.
  • Soutenance :
    • Présentation PowerPoint/Prezi qui accompagnera la soutenance orale.

Ces éléments seront à déposer dans un devoir créé sur TEAMS.

Barème et critères d’évaluation

Le délai de réalisation est court (2 jours), l'avancement du projet sera valorisé compte-tenu du délai.

Soutenance

La soutenance aura lieu le 23/04/2024. L'horaire de passage est dans la partie "Planning".

Le jury sera présidé par l’intervenant.

L’organisation sera la suivante :

  • 10 min de démonstraton technique

  • 20 min de présentation

  • 10 mn de Q&R / restitution

  • 5 mn de délibération

N’oubliez pas de faire un test de bon fonctionnement du projet avant le début de la soutenance.

Grille d’évaluation

Chaque critère est noté sur 5.

Partie Technique

Les risques et les contraintes liés au projet sont identifiés. [Analyse des risques]

Une étude technique est conduite se concentrant sur le fonctionnement de chaque unité du logiciel. La conception est adaptée à l'architecture. [Dossier fonctionnel]

Un cahier des charges techniques particulières ou cahier des charges détaillé est créé accompagné d'une maquette d'un prototype. La solution technique est mise en œuvre en respectant les normes et standards décrits dans le paque. Un dossier de recette ou procès-verbal de réception définitive est élaboré. [Dossier Architecture technique]

Compréhension et mise en application d'une architecture microservices et hexagonale [Architecture logicielle]

Les différents services et composants soient bien intégrés, sécurisés et alignés avec les besoins métier. [Urbanisme SI]

Mise en œuvre et gestion de l'infrastructure, des conteneurs, etc., en alignement avec l'architecture et l'urbanisme définis. [DevOps]

Les certifications sont proposées et préparées. Le plan qualité est rédigé et le processus qualité est piloté. La démarche qualité est respectée. [Démarche qualité]

Organisation du projet

Tenue du planning du projet.

Rapport d'audit de la réalisation du projet. [Point avancement]

Communication et organisation inter-groupe. [Projet abouti et fonctionnel]

Selon le travail fourni 1 point de bonus pourra être attribué.

La note globale sera jugée sur l'aboutissement technique du projet et sa soutenance. Les notes personnelles seront définies sur la qualité du travail de chaque groupe.
